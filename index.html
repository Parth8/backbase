The Architecture

One connector. One API. Three rails.

Grand Central is the middleware. The core does the processing; the connector translates and routes. T24 sends one unified instruction and names the rail. The connector is deliberately dumb — no routing intelligence, no fallback logic. It translates, submits, and returns.

SCT

Standard SEPA credit transfer. Batch, next-day settlement. The forgiving rail.

SCT Inst

Real-time. A hard 10-second deadline. Timeouts become a first-class failure mode.

SWIFT

Cross-border, correspondent chains, compliance holds. Failure semantics unlike SEPA.

Reading the Pattern

Each shape has a root cause — and a consequence

Structured

Documented scheme rejection.

pacs.002 / MT199 with a known code. Ops routes it immediately. The system working.

Partial

T24 code, untranslated.

The connector has no mapping, so it passes the raw core code through. Ops needs the T24 team.

Generic

SWIFT downstream / unhandled.

Correspondent rejection the connector can't parse. Ops has no escalation path at all.

Blank

Timeout. Exception swallowed.

No response captured. Ops doesn't know if the money moved.

The blank case is not just a triage problem — it is a financial one. These are SCT Inst timeouts where payment state is unknown. If ops retries, they risk a duplicate posting. An ambiguous error here can move money twice.

The Question Everyone Asks

“Why didn't QA catch this?”

Because QA did its job. Both Backbase QA and Meridian QA tested against the v2.0 contract. A blank or generic failure is schema-valid under v2.0 — every ambiguous response passes every test. You cannot test for a field the contract never required.

This is a product specification gap, not a QA execution gap. Which makes it mine to own — not a defect to push back on engineering or testing.

Why each bank missed it

Scope, not negligenceApex never ran SCT Inst. Sparkasse never ran SWIFT. Each tested what it had. There was never a shared failure contract to validate across banks — so the gap could only surface where scope finally exceeded it. Here.

The Decision · The Tempting Wrong Answer

Engineering proposes a client-side fix

Let Meridian build a response interpreter on the T24 side that normalises whatever the connector returns. Grand Central changes nothing. It's reasonable, it's fast, it protects the date — and it is exactly what we did at Apex, and at Sparkasse.

It isn't lazy. It's structurally consistent with how this platform has always worked — bank-specific code for bank-specific problems. And that is precisely why I reject it. The third instance of an anti-pattern is the moment to break it, not repeat it.

The Trade-off, Made Explicit

Three options. I'm choosing the un-elegant one.

OptionFixesEng changeRegressionSprint hitBank effort
Client-side workaroundCore errors onlyNoneNoneNoneHigh
Config expansion ✓Core + rail · ~80–90%NoneNoneNoneMedium
Connector code changeAll three layersYesMediumHighLow

The config route doesn't solve origin tagging, and it leaves the three blank timeouts open. But it gives ops 80–90% coverage with no code change, no regression cycle, and no disruption to a fully-loaded team.

I will not push connector changes into a sprint 20 days from go-live. The risk to the date is real, and I need engineering's goodwill for the fix that actually matters. Judgment, not perfection.

The Go-Live Fix · Made Concrete

Config + connector = the integration entity

Two of the three error layers can be fully standardised — rail and connector errors. The core layer cannot: every bank's T24 is different. So the canonical shape is the platform contract; the core mapping is bank-owned configuration. Here is that file.
