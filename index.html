The Architecture

One connector. One API. Three rails.

Grand Central is the middleware. The core does the processing; the connector translates and routes. The core sends one instruction and names the rail. The connector submits it and passes the result back. That return path, the failure path, is where this case lives.

SCT

Standard SEPA transfer. Batch, next-day. The forgiving rail.

SCT Inst

Real-time, a hard 10-second deadline. Timeouts become a real failure mode.

SWIFT

Cross-border. Failure semantics unlike SEPA, and brand new to this connector.

The 47 split into 24 usable and 23 not, and the line between them is one thing: did the connector have a way to translate it?

Where the connector knew the code, ops got a clean answer. Where it did not - a core code with no mapping, a SWIFT code it never standardised, a timeout it never caught - ops got raw, generic, or nothing. It is structural, not random.

The Systemic Signal

Three banks hit the same gap. It never propagates.

Each bank runs its own build of the connector. A patch made for one bank stays in that bank's build unless it is promoted to the shared baseline. It never was. So Meridian inherits none of the earlier fixes, and adds SWIFT on top.

The Decision

Three ways to fix it. Only one is right.

Engineering proposes the fast option: let Meridian build a small interpreter on the T24 side to clean up whatever the connector returns. It works, it protects the date, and it is exactly what Apex and Sparkasse did. That is the reason I say no.

OptionWhat it isRisk to launchVerdict
Bank-side interpreterMeridian cleans up responses on the T24 sideLowNo - repeats the pattern
Change the connector nowRebuild the response logic in codeHighNo - touches tested code
Expand the configMap the missing codes, no code changeLowYes

The bank-side fix makes Meridian own our gap, and leaves the next bank to rediscover it. The third instance of a pattern is where you break it, not repeat it.

I will not push code changes into a tested connector 20 days out. Config gets ops most of the way, with no code risk and no disruption to a fully-loaded team.

The Go-Live Fix

The config and the connector are one unit

The config maps a raw code to three things: where it came from, a canonical code, and a plain-English reason. Rail and connector codes are standard for every bank. Core codes are the bank's own.

gc_error_mapping_config · v3.5
{
  "rail": {          // standard
    "RJCT": {
      "origin": "RAIL",
      "code": "SWIFT_REJECTED",
      "reason": "Rejected by SWIFT. See rawCode."
    }
  },
  "connector": {     // standard
    "TIMEOUT": {
      "origin": "CONNECTOR",
      "code": "RAIL_TIMEOUT",
      "reason": "No response within the window."
    }
  },
  "core": {          // bank-owned
    "AC.CLOSED": {
      "origin": "CORE",
      "code": "ACCOUNT_CLOSED",
      "reason": "Debtor account closed in core."
    }
  },
  "fallback": {      // never blank
    "origin": "UNKNOWN",
    "code": "UNRESOLVED_ERROR",
    "reason": "Unmapped. Raw code kept."
  }
}

Rail · standard

SEPA and SWIFT codes. Finite, regulated, identical for every bank.

Connector · standard

The connector's own codes. The timeout is just one of them now, mapped like any other. No special procedure.

Core · bank-owned

Meridian's T24 codes. Anything unmapped hits the fallback, so a response is never blank.

Where I hold the line: the config says what failed and where. It does not decide whether to retry or who to call. That is the bank's operational policy, not our contract.

The Fix · The Error Packet

Same failure. The packet finally says what and where.

Scenario: SCT payment, the core rejects with AC.CLOSED. Left is what the connector returns today. Right is the v3.5 packet, with every new field labelled by where it comes from.

Today · v3ops cannot use
{
  "paymentId": "MER-PAY-...000211",
  "gcTransactionId": "gc-ccdd3344",
  "status": "FAILED",
  "railName": "SCT",
  "errorCode": "AC.CLOSED",
  "errorMessage": "",
  "failedTimestamp": "2025-03-17T11:11Z"
}

A raw core code, no origin, no meaning, and the message is empty. Ops has to investigate from scratch.

Proposed · v3.5routable on sight
{
  "paymentId": "MER-PAY-...000211",   ← from request
  "gcTransactionId": "gc-ccdd3344",
  "status": "FAILED",
  "origin": "CORE",            ← call graph
  "code": "ACCOUNT_CLOSED",     ← config map
  "reason": "Debtor account closed.", ← config map
  "rawCode": "AC.CLOSED",      ← passthrough
  "railName": "SCT",
  "failedTimestamp": "2025-03-17T11:11Z"
}

Origin from which endpoint the connector called. Code and reason from the config map. Raw code still preserved for the core team.

No invented intelligenceThe packet says what failed and where. It deliberately does not carry retry flags or ops actions - those are the bank's call. Origin is read from the connector's call graph, never guessed from the error text.

The Fix · The Plan

Ship the date. Fix the platform. End the pattern.

Now · for launch

Expand the config to cover the core, rail, and timeout codes. No code change, no regression. UAT cycle 4 targets the 23, not a full re-test. Fully reversible.

In parallel · the real fix

I raise the v3.5 packet with the connector PM as a platform issue. The SA designs it; origin becomes a required field in the contract and the test suite.

After · the pattern ends

Apex and Sparkasse migrate to the baseline. Their orphaned patches become the standard. The next bank inherits it and never sees this.

What I tell Henrik by end of day

"We found the pattern. The connector is behaving to spec; the gap is in how that spec describes failures. We resolve the large majority by configuration, with no change to the code your integration is tested against, and no risk to the date. The deeper fix is already moving at the platform level. The date holds."

It was never broken.
The contract just never described failure.

The connector did exactly what it was told. The fix was not more code - it was teaching the platform to describe how a payment fails, not just how it starts. A connector that fails clearly is worth more than one that succeeds quietly.
